# 代码规范

本文约束 **Agent 与本仓库协作者** 在 `src/`、`web/scripts/` 等可执行代码中的质量与注释要求。目录与 Git 边界见 [hard-constraints.md](./hard-constraints.md)、[../architectuure/repository-layout.md](../architectuure/repository-layout.md)。

## 1. 质量等级与架构

- 以**企业级可维护性**为默认目标：分层清楚、模块边界明确、依赖方向一致，避免将业务、基础设施与展示逻辑搅在一起。
- 新增能力时优先**可扩展**设计：稳定抽象、对修改封闭对扩展开放；避免无谓的硬编码与复制粘贴式分支，除非用户明确要求简单实现。
- **`src/main/java` 粒度**：**一个 `.java` 文件只承担一项可单独表述的任务**；若已无法专一，**先重构**再堆功能。
- **文档同步**：每新增、删除或**实质改变**某 main 源文件职责时，必须更新 [../architectuure/java-modules.md](../architectuure/java-modules.md) 对应行；该文档**不**收录 `src/test/java`。

## 2. 与需求对齐

- 实现须**严格对齐**用户已说明的协议、接口形态与业务规则；文档与对话冲突时以**用户最新明确表述**为准。
- 存在歧义或缺口时**先确认**再写代码，不擅自替用户定产品行为。

## 3. 自动化测试

- **每一个**独立模块或可交付功能在实现时，须对应**至少一个专用测试文件**（独立测试类/脚本文件），不得长期仅靠其它模块的测试顺带覆盖或空壳测试敷衍。
- **Java**：测试类放在 `src/test/java`，包路径与命名与所测生产代码对应（例如 `com.example.foo.BarService` 对应 `…foo.BarServiceTest` 或该模块内约定的 `*Test` 专文件）。
- **`web/scripts/`** 等前端逻辑在引入可执行测试栈后，每个独立模块须具备**对应用例文件**，路径随所选框架约定。
- **全面**：对当前模块职责范围内的主成功路径、关键分支与错误/边界条件做有意识覆盖；对外部云服务等依赖使用替身、内存实现或契约测试，**禁止**在常规测试中依赖真实密钥调用线上接口。
- 无行为的数据载体、纯配置枚举等经用户确认可豁免时，须在交付说明中**写明豁免理由**。

## 4. `config/lua` 下 Lua 配置

- **`return { ... }` 内每一个配置键**均须有**只说明该键**的注释：优先键的**上一行** `-- 说明`，或**行尾** `-- 说明`；不得用仅覆盖全文件的段首注释代替逐项注释。
- 嵌套表中若子键在业务上视为独立可配置项，**每个子键**同样遵守上条。
- 尚无键的占位文件须在文件或空表处注明：后续每增一键必须带专属注释。

## 5. 注释（Java 等源码，必选，量适当）

- **必须**为下列情况编写注释：公共 API、非一眼可读的算法或协议步骤、非常量业务规则、与安全/资源生命周期相关的约定。
- **不必**为下列情况堆注释：语义已由命名表达的赋值、标准库用法的显而易见的单行。
- **风格**：客观陈述，**无情感、无废话**；说明意图、约束、非直观原因或与外部系统的契约，避免复述代码字面含义。

## 6. 语言

- 注释与提交说明默认使用**简体中文**（类名、API、日志 key 等保持项目既有约定，可与英文混用）。

## 7. 维护

- 用户或 Agent 对代码规范有新约定时：**直接修改本文**，并在 [README.md](./README.md) 索引中保持条目存在；不另写迁移文档（与 [documentation-conventions.md](./documentation-conventions.md) §5 一致）。
